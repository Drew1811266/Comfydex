# Comfydex 1.2 Release Checklist

Use this checklist before tagging `v1.2.0`.

## Repository

- Start from a clean worktree or document every intentional local change.
- Confirm the release branch contains only 1.2 Live Bridge productization work.
- Confirm version files report `1.2.0` for Python, Codex plugin manifest, desktop package, package lock, Tauri `Cargo.toml`, Tauri `tauri.conf.json`, and `Cargo.lock`.

## Automated Checks

Run Python tests:

```powershell
python -m pytest -q
```

Run JavaScript syntax checks for the ComfyUI frontend bridge:

```powershell
node --check custom_nodes\comfydex_live_bridge\web\comfydex_live_bridge.js
node --check custom_nodes\comfydex_live_bridge\web\client.js
```

Run PowerShell parser checks:

```powershell
$null = [System.Management.Automation.Language.Parser]::ParseFile("scripts\install_live_bridge.ps1", [ref]$null, [ref]$null)
$null = [System.Management.Automation.Language.Parser]::ParseFile("scripts\live_bridge.ps1", [ref]$null, [ref]$null)
$null = [System.Management.Automation.Language.Parser]::ParseFile("scripts\verify_live_bridge.ps1", [ref]$null, [ref]$null)
```

Run desktop checks from `desktop/`:

```powershell
npm run typecheck
npm run build
cargo check --manifest-path src-tauri\Cargo.toml
```

## Manual Live Bridge Checks

- Run manual ComfyUI restart verification after installing the bridge into a real ComfyUI `custom_nodes` directory.
- Confirm ComfyUI shows the Comfydex frontend extension in its extension list.
- Run no-push verification against the running server.
- Run push verification with a UI workflow and confirm the canvas changes in ComfyUI.
- Confirm push verification with a UI workflow reports a matching acknowledgement request id.
- Confirm the desktop Dashboard and Settings views show Ready after the frontend heartbeat is fresh.
- Confirm Reload client refreshes the browser-side client without a reinstall.
- Confirm Reload backend refreshes bridge runtime code while the backend route remains loaded.

## Release

- Merge the release branch into `main` only after the automated and manual gates pass or the manual limitation is documented.
- Push `main` to GitHub.
- Tag the release as `v1.2.0`.
- Push tag `v1.2.0` to GitHub.
