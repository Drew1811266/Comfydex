# Comfydex 0.8 Gallery And Batch UI Design

## Status

Comfydex `0.7.0` is released on `main`, pushed to GitHub, and tagged as `v0.7.0`.

This document defines the `0.8.0` Gallery And Batch UI release. It builds on the Windows-first Tauri desktop shell from `0.7.0` and the asset library core from `0.6.0`.

## Goal

Comfydex `0.8.0` turns the desktop shell into a practical visual management surface for generated assets and batch records.

The release adds:

- an asset gallery view with dense image/output records,
- asset metadata controls for favorite, rating, tags, and notes,
- safe cleanup planning UI,
- asset comparison UI,
- asset library report generation and preview,
- batch task list and batch detail UI,
- Python desktop bridge operations and Tauri commands for these surfaces.

## Non-Goals

The following remain out of scope for `0.8.0`:

- production installer,
- bundled Python runtime,
- code signing,
- auto-updater,
- full offline packaging,
- actual batch submission UI,
- editing ComfyUI workflow graphs,
- native image thumbnail generation beyond indexed file paths,
- destructive cleanup without an explicit confirmation surface,
- replacing Codex/MCP as the workflow generation and diagnosis engine.

## Architecture

### Python Desktop Bridge

Extend `src/comfydex_mcp/desktop_bridge.py` with operations:

- `update_asset_metadata`
- `plan_asset_cleanup`
- `export_asset_library_report`
- `compare_assets`
- `list_batches`
- `read_batch`

Bridge operations must reuse existing Python modules:

- `assets.update_asset_metadata`
- `assets.plan_asset_cleanup`
- `assets.export_asset_library_report`
- `assets.compare_assets`
- `batches.read_batch_record`
- existing project context/config loading and path safety.

`list_batches` may be implemented as a bridge-local read-only operation over `runs/.batches/*/batch.json` because no public list helper exists yet. It must:

- stay under the configured `runs_dir/.batches` directory,
- skip malformed records with recoverable error summaries,
- sort by `updated_at` descending, then `batch_id`,
- return compact records for UI lists.

### Tauri Commands

Expose commands:

- `update_asset_metadata(payload)`
- `plan_asset_cleanup(payload)`
- `export_asset_library_report(payload)`
- `compare_assets(payload)`
- `list_batches()`
- `read_batch(batchId)`

Rust continues to store only the selected workspace path. It must not reimplement SQLite, cleanup selection, asset comparison, report generation, or batch parsing beyond command dispatch and empty-workspace fallback responses.

### Frontend API

Extend `desktop/src/lib/api.ts` with typed functions:

- `updateAssetMetadata(assetId, patch)`
- `planAssetCleanup(payload)`
- `exportAssetLibraryReport(filters)`
- `compareAssets(leftAssetId, rightAssetId)`
- `listBatches()`
- `readBatch(batchId)`

Browser-only fallback data must remain deterministic and must exercise all major UI states without a native Tauri window.

### Desktop UI

Keep the desktop app as the first screen. Do not add a landing page.

Add a `Batches` navigation item. Keep `Assets` as the primary asset management surface, but upgrade it from table-only to a gallery/workbench view.

#### Assets Gallery

The Assets view should support:

- gallery/table segmented control,
- text search,
- favorite filter,
- minimum rating filter,
- tag filter input,
- selected asset detail,
- favorite toggle,
- rating control,
- tag editing,
- notes editing,
- asset comparison selection,
- cleanup planning.

When an asset has a local path that points to an image-like output, the gallery should attempt to display it. Missing/non-image paths should show a deterministic file tile, not a broken layout.

#### Asset Comparison

The comparison panel should:

- require exactly two selected assets,
- call `compare_assets`,
- show left/right filenames,
- show changed fields,
- handle empty, loading, error, and loaded states.

#### Cleanup UI

The cleanup panel should:

- default to dry-run,
- show candidate count,
- show skipped count and reasons,
- require explicit confirmation before passing `confirm=True`,
- keep deletion out of ordinary gallery interactions.

For `0.8.0`, confirmation may be a checkbox or local UI state; a fuller modal confirmation system is scheduled for `0.9.0`.

#### Reports

The report panel should:

- call `export_asset_library_report`,
- show report path,
- preview markdown text in a scrollable pane,
- never overwrite user-edited report text in the UI.

#### Batches

The Batches view should show:

- batch id,
- label,
- workflow name,
- status,
- run count,
- completed/failed counts,
- created and updated timestamps,
- selected batch detail,
- child run status rows,
- variation parameter preview.

Batch submission remains an MCP/Codex operation in `0.8.0`; the desktop UI reads and inspects batch records.

## Safety

- Cleanup is dry-run by default.
- Confirmed cleanup must call the existing shared cleanup planner.
- The UI must not construct delete paths itself.
- Header values stay redacted.
- Workspace paths remain explicit and local.
- Batch ids must be treated as identifiers, not paths.
- Bridge errors remain stable envelopes.

## Release Gate

Run:

```powershell
python -m pytest tests -q
python scripts\validate_plugin.py
python -m json.tool .codex-plugin\plugin.json > $null
python -m json.tool .mcp.json > $null
npm --prefix desktop run typecheck
npm --prefix desktop run build
cargo check --manifest-path desktop\src-tauri\Cargo.toml
git diff --check
```

Then review:

- bridge path safety,
- cleanup confirmation flow,
- asset metadata payload shape,
- batch id handling,
- frontend loading/error/empty states,
- desktop layout responsiveness,
- documentation/version consistency.
