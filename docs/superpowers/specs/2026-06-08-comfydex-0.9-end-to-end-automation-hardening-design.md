# Comfydex 0.9 End-To-End Automation And Hardening Design

## Status

Comfydex `0.8.0` is released on `main`, pushed to GitHub, and tagged as `v0.8.0`.

This document defines the `0.9.0` End-To-End Automation And Hardening release. It builds on:

- the shared project core and SQLite index from `0.3.0`,
- the workflow generation engine from `0.4.0`,
- the custom node validation loop from `0.5.0`,
- the asset library core from `0.6.0`,
- the Windows-first Tauri desktop shell from `0.7.0`,
- the gallery and batch UI surfaces from `0.8.0`.

## Goal

Comfydex `0.9.0` must provide a reliable one-call path from a natural-language workflow request to saved workflow, submitted run, completion wait, fetched outputs, and project reindexing, while preserving explicit risk confirmation boundaries.

The release also adds release package validation so the final `1.0.0` stage can focus on installer assets, final docs, stability review, and security/path review.

## Scope

### 1. End-To-End Automation Tool

Add a new MCP tool:

```python
async def comfy_generate_run_fetch(
    name: str,
    intent: str,
    parameters: dict[str, Any] | None = None,
    template_id: str | None = None,
    constraints: dict[str, Any] | None = None,
    run_label: str | None = None,
    confirm_risky_actions: bool = False,
    wait_for_completion: bool = True,
    fetch_outputs: bool = True,
    download_outputs: bool = True,
    use_object_info: bool = True,
) -> dict[str, Any]:
    ...
```

The tool must orchestrate these existing capabilities:

1. plan workflow generation,
2. load ComfyUI object metadata when requested,
3. build and validate the generated workflow,
4. evaluate submit policy and confirmation needs,
5. save the workflow only when allowed or explicitly confirmed,
6. submit the workflow,
7. wait for completion when requested,
8. fetch outputs when requested,
9. reindex the project after workflow and output changes,
10. return a structured result envelope with stage, status, policy, run, outputs, and recovery guidance.

The tool must not create a parallel execution stack. It should reuse:

- `plan_workflow_generation`,
- `build_generated_workflow`,
- `evaluate_submit_policy`,
- `save_workflow`,
- `comfy_submit_workflow`,
- `comfy_wait_for_run`,
- `comfy_fetch_outputs`,
- `reindex_project`.

### 2. Automation Result Envelope

The tool returns a deterministic envelope:

```json
{
  "status": "completed",
  "stage": "completed",
  "workflow_name": "generated.json",
  "saved_workflow": "generated.json",
  "policy": {
    "decision": "allowed",
    "requires_confirmation": false,
    "blocked": false,
    "reasons": [],
    "risk_level": "low"
  },
  "generation": {},
  "run": {},
  "outputs": {},
  "index": {},
  "next_actions": []
}
```

Allowed `status` values:

- `blocked`: generation or policy blocked before saving/submitting.
- `requires_confirmation`: generated workflow is valid but risky and `confirm_risky_actions` is false.
- `submitted`: workflow was submitted, but wait/fetch was intentionally skipped.
- `completed`: wait completed and output fetch either completed or was intentionally skipped.
- `failed`: submission, wait, fetch, or reindex failed after a recoverable stage.

Allowed `stage` values:

- `planning`
- `object_info`
- `generation`
- `policy`
- `save`
- `submit`
- `wait`
- `fetch`
- `reindex`
- `completed`

`next_actions` must be concrete strings suitable for Codex to follow, such as:

- `Set confirm_risky_actions=true to overwrite the existing workflow after reviewing policy.reasons.`
- `Run comfy_read_run with the returned run_id.`
- `Run comfy_fetch_outputs after the run reaches completed status.`

### 3. Risk Confirmation Rules

The automation tool may submit automatically only when:

- generation status is valid,
- submit policy decision is `allowed`,
- the request is single-run,
- no workflow overwrite is detected,
- no unknown validation state is detected,
- no missing model/node condition is known,
- no high-cost dimension or step condition is unresolved.

The tool must return `requires_confirmation` without saving or submitting when:

- policy decision is `requires_confirmation`,
- `confirm_risky_actions` is false.

The tool may proceed when:

- policy decision is `requires_confirmation`,
- `confirm_risky_actions` is true,
- generation has a submit-ready workflow,
- policy is not blocked.

The tool must not proceed when:

- policy decision is `blocked`,
- generation has no submit-ready workflow,
- the workflow name is unsafe,
- path safety checks fail,
- the request is batch-like.

### 4. Unknown Object Metadata Hardening

If `use_object_info=True` and ComfyUI object metadata cannot be loaded:

- the tool must not crash during planning,
- it must build using structural validation only,
- policy must include `object_info_unavailable`,
- policy must require confirmation unless another rule blocks the run,
- the response must include the object metadata error text.

This preserves the 1.0 automatic run policy: unknown validation state is not auto-run without explicit confirmation.

### 5. Recovery Behavior

Failures after a run record exists must be recoverable:

- Submit failure should return `failed` with `stage="submit"` and include `run_id` when the lower-level submit tool created a failed run record.
- Wait failure should return `failed` with `stage="wait"` and include the latest run record when readable.
- Fetch failure should return `failed` with `stage="fetch"` and include the latest run record plus guidance to retry `comfy_fetch_outputs`.
- Reindex failure should not hide a successful run. It should return `completed` with an `index_warning` field when output fetch succeeded.

Unsafe path errors may still raise directly because they occur before remote calls and before filesystem writes.

### 6. Release Package Validation

Add a local validation script:

```powershell
python scripts\validate_release_package.py
```

The script must check:

- Python package version, plugin manifest version, desktop npm version, Tauri version, and Cargo version match.
- `.codex-plugin/plugin.json` and `.mcp.json` are valid JSON objects.
- required desktop files exist.
- `README.md` mentions the current version and `comfy_generate_run_fetch`.
- `docs/usage/end-to-end-automation.md` exists and mentions confirmation, wait, fetch, and reindex behavior.
- plugin validation still passes by calling or reusing `validate_plugin`.

The script should return a list of errors from a pure `validate_release_package(root: Path)` function and print a success message from its CLI `main()`.

### 7. Documentation

Update:

- README current version, tool catalog, usage example, and release notes.
- `docs/usage/end-to-end-automation.md`.
- `docs/usage/workflow-generation.md`.
- `skills/comfyui-workflows/SKILL.md`.
- `tests/test_docs_examples.py`.
- version files and plugin manifest.

## Non-Goals

The following remain out of scope for `0.9.0`:

- production installer,
- bundled Python runtime,
- code signing,
- auto-updater,
- offline packaging,
- changing the desktop app into a workflow graph editor,
- automatic batch generation and submission,
- destructive cleanup automation,
- replacing existing low-level MCP tools.

## Architecture

### Server Orchestration

The orchestration should live in `src/comfydex_mcp/server.py` because it coordinates existing MCP tool functions and ComfyUI network operations.

Keep pure policy helpers in `src/comfydex_mcp/generation.py` when they are reusable by both automation and existing workflow generation tools.

Do not introduce another storage layer. Persisted state remains:

- workflow JSON under configured `workflows_dir`,
- run records and workflow snapshots under configured `runs_dir`,
- output files under `runs/<run_id>/outputs`,
- project index under `.comfydex/comfydex.db`.

### Object Info Handling

Use the same `ComfyClient` object metadata access as existing generation tools.

When object metadata loading fails, automation should use structural validation and add:

```json
{
  "object_info_warning": {
    "reason": "object_info_unavailable",
    "error": "..."
  }
}
```

The submit policy must receive `issues=["object_info_unavailable"]` so the decision becomes `requires_confirmation` unless a separate blocking condition exists.

### Policy Hardening

Extend `evaluate_submit_policy` so these issue reasons require confirmation when not blocked:

- `batch_request`
- `object_info_unavailable`
- `unknown_validation`
- `missing_model`
- `missing_node`
- `high_step_count`

Keep these reasons blocking:

- `pixel_count_exceeds_limit`
- `validation_not_submit_ready`

This keeps existing behavior for invalid workflows while making unknown validation explicit.

### Reindexing

After saving a workflow, try to reindex the project. After fetching outputs, try to reindex again with outputs included.

Reindex failures must be reported as warnings. They must not mark an otherwise completed run as failed.

## Release Gate

Run:

```powershell
python -m pytest tests -q
python scripts\validate_plugin.py
python scripts\validate_release_package.py
python -m json.tool .codex-plugin\plugin.json > $null
python -m json.tool .mcp.json > $null
npm --prefix desktop run typecheck
npm --prefix desktop run build
cargo check --manifest-path desktop\src-tauri\Cargo.toml
git diff --check
```

Then review:

- automation policy boundaries,
- path safety before remote calls,
- object metadata unavailable behavior,
- workflow overwrite confirmation,
- run recovery after submit/wait/fetch failures,
- output fetch path safety,
- reindex warnings,
- package validation,
- documentation/version consistency.

## Acceptance Criteria

`0.9.0` can be released only when:

- `comfy_generate_run_fetch` is available as a direct Python function and MCP tool,
- a tested low-risk request can generate, save, submit, wait, fetch, and reindex,
- unconfirmed risky requests do not save or submit,
- object metadata failures require confirmation rather than silently auto-running,
- post-run failures return structured recovery data,
- release package validation passes,
- README, usage docs, and workflow Skill describe the new flow,
- all release gate commands pass on the feature branch and again on merged `main`.
