# Comfydex 0.6 Asset Library Core Design

## Status

Comfydex `0.5.0` is released on `main` and tagged as `v0.5.0`.

This document defines the `0.6.0` asset library core release. It is the next roadmap stage after the custom node complete loop and before the Windows-first Tauri desktop shell.

## Goal

Comfydex `0.6.0` adds a local asset library service on top of the shared project core. A developer should be able to reindex generated outputs, search and filter them, attach tags/ratings/favorite state, write deterministic sidecar metadata, plan safe cleanup, and connect assets back to runs, workflows, reports, and comparisons through MCP tools.

## Non-Goals

The following remain out of scope for `0.6.0`:

- desktop UI,
- visual gallery layout,
- binary image thumbnail rendering with external image libraries,
- cloud sync,
- multi-user annotations,
- model file discovery outside indexed workflow/run records,
- deleting workflow files or run records,
- changing existing run/output directory layout,
- replacing existing `comfy_list_outputs`, `comfy_cleanup_outputs`, `comfy_export_run_report`, or `comfy_compare_runs`.

## Existing Starting Point

The shared core already has:

- project context and `.comfydex/comfydex.db`,
- schema version `1`,
- `workflow_records`,
- `run_records`,
- `output_records`,
- `batch_records`,
- `index_errors`,
- `comfy_project_status`,
- `comfy_reindex_project`.

Output listing and cleanup already exist in `outputs.py`, while run reports and run comparisons already exist in `reports.py` and `diagnostics.py`.

The gap is that `output_records` is a compatibility index, not an asset library. It does not preserve user annotations, expose search filters, store prompt/model summary fields, write sidecar metadata, or provide asset-level cleanup planning.

## Architecture

### Core Module

Add `src/comfydex_mcp/assets.py` as the asset library service layer.

Responsibilities:

- convert indexed output rows into asset rows,
- preserve user annotations across reindex,
- search assets through structured filters,
- update asset annotations,
- write deterministic sidecar metadata,
- build cleanup plans by asset id or search filter,
- produce asset comparison summaries for two asset ids,
- produce a markdown asset library report.

`assets.py` should use public shared-core database helpers and path safety helpers. It must not call private MCP server internals.

### SQLite Schema Version 2

Bump the schema to version `2` and add a migration named `asset_library_core`.

Create `asset_records`:

```sql
CREATE TABLE IF NOT EXISTS asset_records (
  asset_id TEXT PRIMARY KEY,
  output_id TEXT NOT NULL UNIQUE,
  run_id TEXT NOT NULL,
  workflow_name TEXT,
  status TEXT NOT NULL,
  prompt_text TEXT NOT NULL,
  model_references_json TEXT NOT NULL,
  path TEXT NOT NULL,
  filename TEXT NOT NULL,
  type TEXT NOT NULL,
  subfolder TEXT NOT NULL,
  size_bytes INTEGER NOT NULL,
  modified_time REAL NOT NULL,
  content_hash TEXT NOT NULL,
  sidecar_path TEXT,
  thumbnail_path TEXT,
  thumbnail_status TEXT NOT NULL,
  tags_json TEXT NOT NULL,
  rating INTEGER,
  favorite INTEGER NOT NULL,
  notes TEXT NOT NULL,
  indexed_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  FOREIGN KEY (run_id) REFERENCES run_records(run_id) ON DELETE CASCADE,
  FOREIGN KEY (output_id) REFERENCES output_records(output_id) ON DELETE CASCADE
);
```

Create indexes:

- `idx_asset_records_run_id`
- `idx_asset_records_workflow_name`
- `idx_asset_records_status`
- `idx_asset_records_modified_time`
- `idx_asset_records_favorite`
- `idx_asset_records_rating`

The migration must preserve existing version `1` databases and must not delete existing records.

### Asset Identity

Use `asset_id = output_id`.

`output_id` is already stable for a given `run_id` and resolved file path. This keeps asset identity compatible with the existing project index and avoids a second identity scheme.

If a file is moved or deleted, the next reindex removes the stale asset row. User annotations are preserved only while the same `asset_id` exists. This is acceptable for `0.6.0` because Comfydex does not yet manage file relocation.

### Annotation Preservation

Asset user fields are:

- `tags_json`,
- `rating`,
- `favorite`,
- `notes`,
- `updated_at`.

When reindexing assets, physical fields are refreshed from `output_records`, `run_records`, and workflow snapshots. User fields are copied from the previous `asset_records` row when the `asset_id` already exists.

Validation:

- tags are a sorted unique list of non-empty strings, maximum 40 tags,
- each tag is trimmed, whitespace-collapsed, and limited to 40 characters,
- rating is `null` or an integer from `1` to `5`,
- favorite is boolean,
- notes are whitespace-collapsed and limited to 1000 characters.

### Prompt And Model Metadata

For each asset, derive metadata from existing records:

- `workflow_name` from the run record,
- `status` from the run record,
- `prompt_text` from workflow snapshot text inputs when available,
- `model_references_json` from workflow analysis when available.

Prompt extraction must be deterministic and conservative:

- inspect workflow snapshot node inputs,
- collect string values from common prompt keys such as `text`, `positive`, `positive_prompt`, `negative`, and `negative_prompt`,
- collapse whitespace,
- deduplicate,
- limit final prompt text to 1000 characters.

If no prompt text is found, store an empty string.

### Sidecar Metadata

`write_asset_sidecars(context, asset_ids=None)` writes JSON files under:

```text
.comfydex/assets/sidecars/<asset_id>.json
```

Each sidecar includes:

- asset identity,
- run id,
- workflow name,
- status,
- output path,
- filename,
- type,
- size,
- modified time,
- content hash,
- prompt text,
- model references,
- tags,
- rating,
- favorite,
- notes.

Writes must be atomic through a temporary file in the sidecar directory.

Sidecars are metadata only. `0.6.0` records `thumbnail_status` as:

- `not_supported` for files whose extension is not a common image extension,
- `pending` for common image extensions when no thumbnail file exists,
- `available` when a thumbnail file is already present at the expected path.

Binary thumbnail rendering is deferred to a later UI/gallery stage because the current package has no image processing dependency.

### Search And Filters

Add `search_assets(context, filters)` with stable deterministic ordering.

Supported filters:

- text query across filename, workflow name, prompt text, notes, tags, and model references,
- `run_id`,
- `workflow_name`,
- `status`,
- output `type`,
- tags requiring all selected tags,
- `favorite`,
- `min_rating`,
- `date_from`,
- `date_to`,
- `limit`,
- `offset`.

Default order:

1. `favorite DESC`
2. `rating DESC NULLS LAST`
3. `modified_time DESC`
4. `filename ASC`

Results must include `total`, `limit`, `offset`, and `assets`.

### Cleanup Planning

Add `plan_asset_cleanup(context, filters=None, asset_ids=None, confirm=False)`.

Behavior:

- selects candidates from `asset_records`,
- rejects unsafe paths that escape the configured runs directory,
- dry-run by default,
- when `confirm=False`, returns candidates and deletes nothing,
- when `confirm=True`, deletes only regular files that still match safe path checks immediately before deletion,
- never deletes run directories, run records, workflow snapshots, sidecars, reports, or database rows directly,
- after confirmed deletion, reindexing can remove stale asset rows.

This mirrors the safety model of `cleanup_outputs` while adding asset ids and search filters.

### Reports And Comparisons

Add:

- `export_asset_library_report(context, filters=None)`,
- `compare_assets(context, left_asset_id, right_asset_id)`.

The asset library report writes:

```text
.comfydex/reports/asset-library-report.md
```

It includes filter summary, counts, favorite/rating summary, and a deterministic asset table.

Asset comparison returns:

- asset identity and paths,
- run ids,
- workflow names,
- status comparison,
- prompt text comparison,
- model reference comparison,
- file size comparison,
- tag/rating/favorite comparison.

This does not replace `comfy_compare_runs`; it gives the future desktop gallery a direct asset-to-asset comparison surface.

## MCP Tools

Add these tools:

### `comfy_reindex_assets(include_sidecars=False)`

Runs the project reindex and rebuilds `asset_records`. If `include_sidecars=True`, writes sidecar metadata after reindex.

Returns:

- status,
- database path,
- schema version,
- counts,
- asset count,
- sidecar count,
- errors.

### `comfy_search_assets(...)`

Searches the asset library using the filters listed above.

Returns:

- total,
- limit,
- offset,
- assets.

### `comfy_update_asset_metadata(asset_id, tags=None, rating=None, favorite=None, notes=None)`

Updates user annotations for one asset and returns the updated asset row.

### `comfy_write_asset_sidecars(asset_ids=None)`

Writes sidecar JSON files for selected assets or every indexed asset.

### `comfy_plan_asset_cleanup(filters=None, asset_ids=None, confirm=False)`

Builds a dry-run cleanup plan by default and can delete confirmed candidate files.

### `comfy_export_asset_library_report(filters=None)`

Writes the deterministic asset library markdown report.

### `comfy_compare_assets(left_asset_id, right_asset_id)`

Compares two asset rows.

## Backward Compatibility

Existing tools remain available and unchanged:

- `comfy_list_outputs`,
- `comfy_cleanup_outputs`,
- `comfy_export_run_report`,
- `comfy_compare_runs`,
- `comfy_project_status`,
- `comfy_reindex_project`.

`comfy_reindex_project` should migrate the database to schema version `2` and populate `asset_records` only through the new asset indexing service or through a safe shared helper. Existing project status counts may add an `assets` count without removing previous count keys.

Existing version `1` databases must migrate without requiring manual deletion.

## Path Safety

All asset paths must remain under the configured runs directory. The implementation must reject:

- traversal asset ids,
- absolute path injection,
- symlink/reparse redirection,
- deleted files during confirmed cleanup,
- non-file cleanup candidates,
- sidecar paths outside `.comfydex/assets/sidecars/`.

Cleanup must re-check every file immediately before deletion.

## Error Handling

Recoverable indexing errors must be recorded in `index_errors` and returned by reindex tools.

Search and metadata tools should raise `ValueError` for invalid user inputs:

- unsafe asset id,
- unknown asset id,
- invalid rating,
- invalid limit/offset,
- invalid date range,
- invalid tags.

Sidecar write failures for individual assets should be reported per asset without aborting the entire batch.
