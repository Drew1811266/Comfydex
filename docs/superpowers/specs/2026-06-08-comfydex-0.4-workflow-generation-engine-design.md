# Comfydex 0.4 Workflow Generation Engine Design

Date: 2026-06-08
Version target: `0.4.0`
Parent roadmap: `docs/superpowers/specs/2026-06-08-comfydex-1.0-roadmap-design.md`
Previous release: `0.3.0`

## Status

Comfydex `0.3.0` is released on `main` and tagged as `v0.3.0`.

This document defines the `0.4.0` release scope. Implementation must follow a
separate plan under `docs/superpowers/plans/` and must pass this design's release
gate before tagging `v0.4.0`.

## Context

Comfydex already has a first-pass builder:

- `templates.py` stores a fixed catalog of workflow templates.
- `builder.py` converts a template plan into API prompt JSON.
- `validation.py` validates API workflow shape and object-info compatibility.
- `server.py` exposes `comfy_build_workflow_plan`,
  `comfy_explain_workflow_plan`, and `comfy_build_workflow`.

That is enough for simple template-driven generation, but it is not yet a
workflow generation engine. The 1.0 roadmap requires Codex to generate, validate,
repair, and safely run non-trivial workflows from natural-language requests.

`0.4.0` builds the engine layer. Codex remains the language model that interprets
open-ended user intent. The plugin must provide deterministic planning,
assembly, validation, repair, and risk-policy tools that Codex can call safely.

## Goal

Deliver a deterministic workflow generation engine that turns natural-language
intent plus structured constraints into a validated workflow generation plan,
builds or repairs API prompt JSON, classifies submit risk, and exposes a clear
submit policy.

## Product Outcome

After `0.4.0`, a developer can ask Codex to create a ComfyUI workflow and Codex
can use Comfydex to:

- normalize the request into a structured generation plan,
- choose and explain candidate templates,
- identify required missing inputs,
- build a draft API workflow,
- validate the workflow against API shape and ComfyUI object metadata,
- apply safe mechanical repairs,
- save a generated workflow only when policy allows,
- classify whether submission is allowed, requires confirmation, or is blocked,
- explain every unresolved gap in machine-readable and human-readable form.

## Non-Goals

`0.4.0` must not implement:

- a cloud or external LLM service inside the plugin,
- arbitrary graph synthesis for every ComfyUI node,
- desktop UI surfaces,
- custom node package contract tests beyond existing `0.2.0` tools,
- asset gallery, thumbnails, tags, ratings, or favorites,
- batch UI or large-scale queue orchestration,
- automatic high-risk execution without confirmation,
- guaranteed conversion of every UI workflow into API prompt JSON.

## Design Principles

1. Deterministic engine, Codex-facing language layer.
   Codex interprets natural language; Comfydex validates and executes structured
   plans deterministically.

2. Preserve existing tool compatibility.
   Existing builder tools remain available. `0.4.0` may extend their return
   payloads, but it must not remove current fields that tests or users depend on.

3. Validate before save and submit.
   Generated workflows must expose validation status and submit policy before
   Codex attempts to run them.

4. Repair only mechanical issues.
   The engine may fill safe defaults, normalize numeric bounds, rename invalid
   output prefixes, and produce draft workflows. It must not invent unavailable
   node classes or pretend missing object metadata is valid.

5. Risk policy is explicit.
   Normal validated single-run workflows can be marked `allowed`. Missing nodes,
   missing models, unknown required inputs, overwrites, batch runs, high-cost
   settings, cleanup, or unknown validation must require confirmation or block.

6. Shared-core ready.
   Generated workflow records should remain compatible with the `0.3.0` project
   index and should be reindexable from workflow JSON and metadata.

## Generation Model

The engine introduces a stable generation plan shape:

```json
{
  "intent": "Create an SDXL text-to-image workflow for a city at night",
  "mode": "text-to-image",
  "selected_template_id": "sdxl-text-to-image",
  "candidate_templates": [
    {
      "template_id": "sdxl-text-to-image",
      "score": 91,
      "reasons": ["intent mentions sdxl", "intent asks for text-to-image"]
    }
  ],
  "parameters": {
    "checkpoint_name": "sdxl.safetensors",
    "positive_prompt": "city at night",
    "negative_prompt": "",
    "width": 1024,
    "height": 1024,
    "steps": 30,
    "cfg": 7.0,
    "seed": -1
  },
  "constraints": {
    "allow_overwrite": false,
    "allow_batch": false,
    "max_steps": 60,
    "max_pixels": 1048576
  },
  "missing_information": [],
  "assumptions": [
    "The selected checkpoint is available in ComfyUI."
  ]
}
```

The plan must be JSON-compatible and safe for Codex to inspect or edit before
building.

## Template Selection

`0.4.0` keeps the existing template catalog but upgrades selection from the
current first-match keyword check to scored candidate selection.

Selection signals:

- explicit `template_id`,
- intent keywords,
- mode keywords such as text-to-image, image-to-image, upscale, SDXL, LoRA, and
  ControlNet,
- provided parameters that match template required inputs,
- template tags.

Return at least the top three candidates when available. The selected template is
the highest-scoring candidate unless the caller passed `template_id`.

## Parameter Normalization

The engine should normalize common parameters:

- `width`, `height`, `steps`, and `seed` to integers,
- `cfg`, `denoise`, `strength_model`, `strength_clip`, and
  `control_strength` to floats,
- prompt fields to strings,
- `filename_prefix` to a safe simple string,
- missing default values from the selected template.

Invalid numeric values should produce a repairable issue when a safe clamp exists
and a blocking issue when the value cannot be interpreted.

Default safety constraints:

```json
{
  "allow_overwrite": false,
  "allow_batch": false,
  "max_steps": 60,
  "max_pixels": 1048576
}
```

## Validation And Repair Loop

The engine should expose a validation/repair result:

```json
{
  "status": "valid",
  "submit_ready": true,
  "workflow": {},
  "draft_workflow": null,
  "validation": {},
  "repairs": [
    {
      "kind": "clamped_steps",
      "path": "parameters.steps",
      "before": 150,
      "after": 60
    }
  ],
  "gaps": [],
  "policy": {}
}
```

Allowed repair types for `0.4.0`:

- clamp `steps` to `constraints.max_steps`,
- reject dimensions whose pixel count exceeds `constraints.max_pixels`,
- fill missing optional template defaults,
- sanitize `filename_prefix`,
- normalize seed `-1` as random seed sentinel,
- save invalid generated workflows only as drafts when explicitly requested.

Disallowed repairs:

- inventing unknown node classes,
- guessing unavailable model filenames,
- ignoring object-info validation errors,
- replacing one workflow family with another without reporting the change.

## Submit Policy

The submit policy result shape:

```json
{
  "decision": "allowed",
  "requires_confirmation": false,
  "blocked": false,
  "reasons": [],
  "risk_level": "low"
}
```

Decisions:

- `allowed`: workflow is valid, submit-ready, single-run, and within risk
  thresholds.
- `requires_confirmation`: workflow may be valid, but one or more high-risk
  conditions require explicit user confirmation.
- `blocked`: workflow is not submit-ready or has unresolved missing nodes,
  missing models, invalid API shape, or unsafe file overwrite.

High-risk conditions:

- missing model reference,
- missing node type,
- unresolved required input,
- existing target workflow would be overwritten and `allow_overwrite` is false,
- batch or multi-run request,
- `steps > max_steps`,
- `width * height > max_pixels`,
- unknown validation state,
- output cleanup or deletion request.

## MCP Tools

### `comfy_plan_workflow_generation`

Inputs:

```json
{
  "intent": "string",
  "parameters": {},
  "template_id": null,
  "constraints": {}
}
```

Returns the generation plan with candidate templates, normalized parameters,
missing information, assumptions, and unresolved issues.

This tool does not call ComfyUI.

### `comfy_generate_workflow`

Inputs:

```json
{
  "name": "generated.json",
  "intent": "string",
  "parameters": {},
  "template_id": null,
  "constraints": {},
  "allow_draft": false,
  "use_object_info": true
}
```

Returns the generation result with workflow or draft workflow, validation,
repairs, gaps, and submit policy.

If a valid submit-ready workflow is saved, metadata source should be
`generated`. If a draft is saved because `allow_draft` is true, metadata source
should be `generated` and validation status should show the unresolved state.

### `comfy_evaluate_submit_policy`

Inputs:

```json
{
  "name": "workflow.json",
  "constraints": {}
}
```

Reads an existing workflow and returns submit policy without submitting it.

This tool may call ComfyUI object info when available through existing config
timeouts, but must return a clear `unknown_validation` reason if object info
cannot be read.

## Existing Tool Compatibility

Existing tools remain available:

- `comfy_build_workflow_plan`
- `comfy_explain_workflow_plan`
- `comfy_build_workflow`

Compatibility behavior:

- `comfy_build_workflow_plan` may delegate to the new planning engine but must
  keep returning `intent`, `template`, `required_nodes`, `parameters`,
  `assumptions`, and `missing_information`.
- `comfy_build_workflow` may delegate to `comfy_generate_workflow` but must keep
  returning `status`, `submit_ready`, `workflow`, `validation`, `plan`, `gaps`,
  and `missing_information`.

## Project Index Integration

When `comfy_generate_workflow` saves a workflow or draft, the saved JSON and
metadata must remain compatible with `0.3.0` reindexing.

`0.4.0` may call the project index reindexer after saving generated workflows.
If index refresh fails, the generation result should include an index warning and
must preserve the saved workflow file.

## Testing Requirements

Add tests for:

- scored template candidate selection,
- explicit template override,
- parameter normalization,
- missing required information,
- safe repair records,
- blocking dimension or step risk,
- submit policy allowed/requires-confirmation/blocked decisions,
- generated workflow save metadata,
- draft saving with unresolved validation,
- existing builder tool compatibility,
- MCP tool behavior,
- documentation/version consistency.

The release gate must run:

```powershell
python -m pytest tests -q
python scripts\validate_plugin.py
python -m json.tool .codex-plugin/plugin.json > $null
python -m json.tool .mcp.json > $null
git diff --check
```

## Review Gates

### Spec Compliance Review

Verify:

- generation plan model exists,
- scored candidate selection exists,
- parameter normalization exists,
- validation/repair loop exists,
- submit policy exists,
- MCP tools are exposed,
- existing builder tools remain compatible,
- docs and Skill are updated.

Critical or Important findings block release.

### Code Quality Review

Review for:

- nondeterministic template selection,
- unsafe filename or workflow overwrite behavior,
- validation bypasses,
- broad exception handling that hides object-info failures,
- broken existing builder return contracts,
- missing tests for high-risk policy paths.

Critical or Important findings block release.

### Release Metadata Review

Verify:

- `README.md` says `0.4.0`,
- `pyproject.toml` says `0.4.0`,
- `.codex-plugin/plugin.json` says `0.4.0`,
- `src/comfydex_mcp/__init__.py` says `0.4.0`,
- `tests/test_version.py` expects `0.4.0`,
- release notes mention workflow generation engine,
- tag `v0.4.0` is created only after tests and reviews pass.

## Milestones

### Milestone 1: Generation Plan Model

Deliverables:

- plan model helpers,
- scored template candidate selection,
- parameter and constraint normalization,
- compatibility adapter for existing plan return shape.

Exit criteria:

- plan/model tests pass,
- existing template tests pass.

### Milestone 2: Validation, Repair, And Policy

Deliverables:

- repair result model,
- safe numeric and filename repairs,
- submit policy classifier,
- tests for allowed, confirmation, and blocked decisions.

Exit criteria:

- validation and policy tests pass,
- existing workflow validation tests pass.

### Milestone 3: MCP Tool Integration

Deliverables:

- `comfy_plan_workflow_generation`,
- `comfy_generate_workflow`,
- `comfy_evaluate_submit_policy`,
- compatibility for existing builder tools.

Exit criteria:

- server tool tests pass,
- no existing MCP tool tests regress.

### Milestone 4: Documentation And Release Metadata

Deliverables:

- README update,
- workflow generation usage guide,
- Skill update,
- version bump to `0.4.0`.

Exit criteria:

- docs tests pass,
- version tests pass,
- plugin validation passes.

### Milestone 5: Release Gate

Deliverables:

- full tests,
- plugin validation,
- JSON manifest validation,
- diff check,
- review gates,
- merge/push/tag.

Exit criteria:

- `main` is pushed,
- tag `v0.4.0` is pushed,
- next version `0.5.0` planning starts.

## Release Criteria

`0.4.0` can be released only when:

- all milestones pass,
- full tests pass,
- plugin validation and JSON manifest validation pass,
- `git diff --check` is clean,
- review gates have no unresolved Critical or Important findings,
- release metadata is consistent,
- `main` is pushed to GitHub,
- `v0.4.0` is created and pushed.
