# Comfydex 0.5 Custom Node Complete Loop Design

Date: 2026-06-08
Version target: `0.5.0`
Parent roadmap: `docs/superpowers/specs/2026-06-08-comfydex-1.0-roadmap-design.md`
Previous release: `0.4.0`

## Status

Comfydex `0.4.0` is released on `main` and tagged as `v0.4.0`.

This document defines the `0.5.0` release scope. Implementation must follow a
separate plan under `docs/superpowers/plans/` and must pass this design's release
gate before tagging `v0.5.0`.

## Context

Comfydex already supports custom node development through:

- workspace-local package scaffolding,
- static package inspection,
- `NODE_CLASS_MAPPINGS` validation,
- class contract validation,
- isolated import checks,
- deterministic node documentation.

These tools catch structural issues, but the developer loop is still incomplete.
Developers need example inputs, runnable contract checks, common ComfyUI type
validation, richer import diagnostics, and actionable repair guidance.

## Goal

Deliver a complete local custom node development loop for a single package:
inspect, validate, generate example inputs, run deterministic contract checks,
summarize import failures, and return repair guidance without modifying the
developer's node code.

## Product Outcome

After `0.5.0`, Codex can help a developer iterate on a custom node package by
calling Comfydex tools that:

- inspect package mappings,
- validate classes and common input/output types,
- generate example input payloads from `INPUT_TYPES`,
- run a class method contract check in an isolated subprocess,
- verify return tuple shape and output count,
- summarize import errors with traceback location hints,
- produce structured repair guidance for mapping, class, import, and contract
  failures.

## Non-Goals

`0.5.0` must not implement:

- installing packages into a user's ComfyUI installation,
- starting or controlling a live ComfyUI process,
- executing arbitrary graph workflows,
- image/latent tensor synthesis for complex nodes,
- automatic code modification,
- remote or cloud package validation,
- desktop UI.

## Design Principles

1. Static checks first.
   Use AST inspection wherever possible. Execute code only in isolated contract
   tests or import checks.

2. Workspace-local by default.
   Package writes remain inside `custom_nodes/<package_name>`.

3. Bounded execution.
   Contract tests must use subprocess isolation, timeouts, bounded output, and
   Windows process cleanup behavior consistent with import checks.

4. Common types are explicit.
   `INT`, `FLOAT`, `STRING`, `BOOLEAN`, and common scalar/list options get
   generated examples. Complex data types such as `IMAGE`, `LATENT`, `MODEL`,
   `CLIP`, `VAE`, and `CONDITIONING` are reported as unsupported for automatic
   example generation unless a default value is present.

5. No false success.
   If required examples cannot be generated, the contract test is `blocked`.
   If execution fails, the result is `failed` with diagnostics and repair
   guidance.

## Example Input Generation

Add a helper that reads `node_class_details(package_dir, class_name)` and returns
a deterministic example payload.

Result shape:

```json
{
  "status": "generated",
  "class_name": "SimpleMathNode",
  "function": "run",
  "inputs": {
    "a": 1,
    "b": 1
  },
  "unsupported_inputs": [],
  "warnings": []
}
```

Unsupported required inputs produce:

```json
{
  "status": "blocked",
  "unsupported_inputs": [
    {"name": "image", "type": "IMAGE", "reason": "unsupported_example_type"}
  ]
}
```

Supported examples:

- `INT`: default option if present, otherwise `1`,
- `FLOAT`: default option if present, otherwise `1.0`,
- `STRING`: default option if present, otherwise `"example"`,
- `BOOLEAN` or `BOOL`: default option if present, otherwise `true`,
- enum/list choices: first choice,
- `COMBO`-style list specs: first string choice.

## Contract Tests

Add a contract runner that:

1. validates the package mapping,
2. validates the requested class,
3. generates example inputs,
4. imports the package in an isolated subprocess,
5. instantiates the mapped class,
6. calls the class `FUNCTION` with generated inputs,
7. verifies the result is a tuple,
8. verifies tuple length matches `RETURN_TYPES`,
9. returns stdout/stderr, diagnostics, examples, and repair guidance.

Result statuses:

- `passed`: import, execution, tuple shape, and output count all pass.
- `blocked`: static validation or example generation prevents execution.
- `failed`: import or execution fails.

The runner must not execute in the main MCP process.

## Type Validation

Extend class validation with common type checks:

- `RETURN_TYPES` entries must be known or custom strings.
- `INPUT_TYPES` sections must be dictionaries.
- required/optional input specs must be string, tuple, list, or option tuple
  compatible with ComfyUI conventions.
- common scalar specs should be example-generatable.

Validation should not reject custom types solely because they are not built-in;
it should mark example generation unsupported when required runtime objects are
needed.

## Import Diagnostics

Enhance import check results with:

```json
{
  "diagnostics": {
    "summary": "Import failed: ModuleNotFoundError: No module named 'x'",
    "exception_type": "ModuleNotFoundError",
    "message": "No module named 'x'",
    "file": "nodes.py",
    "line": 3
  }
}
```

Diagnostics must be derived from stderr/stdout text and should stay conservative.
If parsing fails, return a short summary and leave location fields null.

## Repair Guidance

Add package-level guidance that aggregates:

- mapping validation errors,
- class validation errors for mapped classes,
- import check diagnostics,
- contract test result when requested.

Guidance shape:

```json
{
  "status": "needs_work",
  "actions": [
    {
      "priority": "high",
      "reason": "missing_display_name",
      "message": "Add NODE_DISPLAY_NAME_MAPPINGS for GoodNode."
    }
  ]
}
```

Guidance status:

- `ready`: no actions,
- `needs_work`: actions exist,
- `blocked`: package cannot be inspected.

## MCP Tools

### `comfy_generate_node_examples`

Inputs:

```json
{
  "package_name": "simple_math",
  "class_name": "SimpleMathNode"
}
```

Returns the generated example input payload.

### `comfy_run_node_contract_tests`

Inputs:

```json
{
  "package_name": "simple_math",
  "class_name": "SimpleMathNode",
  "timeout_seconds": 5,
  "max_output_bytes": 20000
}
```

Returns contract execution status, examples, diagnostics, stdout/stderr, and
repair guidance.

### `comfy_custom_node_repair_guidance`

Inputs:

```json
{
  "package_name": "simple_math"
}
```

Returns package-level repair guidance without modifying files.

- `comfy_generate_node_docs`

`comfy_check_node_imports` may add `diagnostics`, but existing top-level fields
must remain stable.

## Testing Requirements

Add tests for:

- example generation for scalar inputs,
- blocked example generation for unsupported required runtime inputs,
- contract test pass for scaffolded package,
- contract test blocked for invalid class contract,
- contract test failed for runtime exception,
- tuple output count mismatch,
- import diagnostics extraction,
- repair guidance for mapping and class errors,
- MCP tools,
- docs/version consistency.

The release gate must run:

```powershell
python -m pytest tests -q
python scripts\validate_plugin.py
python -m json.tool .codex-plugin/plugin.json > $null
python -m json.tool .mcp.json > $null
git diff --check
```

## Review Gates

### Spec Compliance Review

Verify:

- example generation exists,
- contract tests run isolated,
- common type validation exists,
- import diagnostics exist,
- repair guidance exists,
- MCP tools are exposed,
- existing custom node tools remain compatible,
- docs and Skill are updated.

Critical or Important findings block release.

### Code Quality Review

Review for:

- unsafe package paths,
- contract execution in the main MCP process,
- timeout or process cleanup gaps,
- unbounded stdout/stderr,
- broad exception handling that hides failures,
- misleading repair guidance,
- missing tests for unsupported required runtime inputs.

Critical or Important findings block release.

### Release Metadata Review

Verify:

- `README.md` says `0.5.0`,
- `pyproject.toml` says `0.5.0`,
- `.codex-plugin/plugin.json` says `0.5.0`,
- `src/comfydex_mcp/__init__.py` says `0.5.0`,
- `tests/test_version.py` expects `0.5.0`,
- release notes mention custom node complete loop,
- tag `v0.5.0` is created only after tests and reviews pass.
