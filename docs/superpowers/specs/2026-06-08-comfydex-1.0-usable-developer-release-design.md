# Comfydex 1.0 Usable Developer Release Design

## Status

Comfydex `0.9.0` is released on `main`, pushed to GitHub, and tagged as `v0.9.0`.

This document defines the final `1.0.0` release. The 1.0 release closes the roadmap by turning the accumulated MCP server, shared core, desktop shell, asset library, automation, and custom-node loop into a Windows-first local developer release that can be installed, verified, reviewed, and maintained.

## Goal

Comfydex `1.0.0` must be a usable developer release for a Windows user who runs ComfyUI locally and wants Codex to help build, validate, run, debug, and manage ComfyUI workflows.

The release must provide:

- final version and manifest metadata,
- local Windows install instructions,
- a local install helper script,
- final release checklist,
- security/path review documentation,
- final README and usage docs,
- package validation that enforces 1.0 release assets,
- full regression verification and release tag.

## Scope

### 1. Windows Local Install Asset

Add:

```text
scripts/install_windows.ps1
```

The script should:

- resolve the repository root,
- check Python availability,
- install the Python package in editable local development mode,
- optionally install desktop npm dependencies,
- optionally run verification commands,
- avoid deleting files,
- avoid modifying ComfyUI installation files,
- print clear next steps for Codex plugin discovery and ComfyUI connection checks.

The script is a local developer install helper. It is not a signed production installer and must not claim to be one.

### 2. Release Documentation

Add:

```text
docs/release/windows-install.md
docs/release/1.0-release-checklist.md
docs/release/security-path-review.md
```

`windows-install.md` must describe:

- prerequisites,
- local clone path,
- Python editable install,
- optional desktop dependency install,
- plugin discovery refresh,
- local ComfyUI connection check,
- release package validation.

`1.0-release-checklist.md` must describe:

- required release gate commands,
- merge and tag checks,
- remote GitHub verification,
- known 1.0 non-goals.

`security-path-review.md` must summarize:

- workflow path safety,
- run/output path safety,
- asset cleanup confirmation,
- header redaction,
- custom node workspace boundary,
- desktop bridge workspace boundary,
- remaining 1.0 limitations.

### 3. Release Package Validation

Extend `scripts/validate_release_package.py` so version `1.0.0` and later require:

- `scripts/install_windows.ps1`,
- `docs/release/windows-install.md`,
- `docs/release/1.0-release-checklist.md`,
- `docs/release/security-path-review.md`,
- README mentions `Usable Developer Release`,
- README mentions `scripts/install_windows.ps1`,
- release docs contain install, verification, and safety markers.

This keeps final release assets testable and prevents future version bumps from accidentally dropping them.

### 4. Final Docs And Metadata

Update:

- README current version and release notes,
- `docs/usage/end-to-end-automation.md` where needed,
- `docs/usage/desktop-app.md` boundaries where needed,
- workflow Skill final 1.0 behavior summary,
- version files to `1.0.0`,
- tests for docs and version consistency.

### 5. Final Verification And Release

The final release gate must run:

```powershell
python -m pytest tests -q
python scripts\validate_plugin.py
python scripts\validate_release_package.py
python -m json.tool .codex-plugin\plugin.json > $null
python -m json.tool .mcp.json > $null
npm --prefix desktop run typecheck
npm --prefix desktop run build
cargo check --manifest-path desktop\src-tauri\Cargo.toml
git diff --check
```

Then:

- review spec compliance,
- review path safety,
- review release docs,
- merge to `main`,
- rerun the release gate on `main`,
- push `main`,
- tag and push `v1.0.0`,
- verify remote `main` and `v1.0.0` point to the same commit.

## Non-Goals

The following remain out of scope for `1.0.0`:

- signed Windows installer,
- bundled Python runtime,
- auto-updater,
- offline dependency bundle,
- team collaboration,
- cloud server fleet management,
- ComfyUI frontend extension,
- modifying ComfyUI installation files,
- marketplace/model distribution.

## Acceptance Criteria

`1.0.0` can be released only when:

- all roadmap versions `0.3.0` through `0.9.0` have been released and tagged,
- `scripts/install_windows.ps1` exists and is documented,
- release docs cover install, checklist, and security/path review,
- `validate_release_package.py` enforces 1.0 release assets,
- all version metadata is `1.0.0`,
- README and Skills describe final 1.0 behavior accurately,
- all release gate commands pass on the feature branch and merged `main`,
- GitHub remote `main` and `v1.0.0` point to the same merge commit.
