# Comfydex 0.7 Tauri Desktop App Shell Design

## Status

Comfydex `0.6.0` is released on `main` and tagged as `v0.6.0`.

This document defines the `0.7.0` Windows-first Tauri desktop app shell release. It is the next roadmap stage after the asset library core and before the gallery/batch UI release.

## Goal

Comfydex `0.7.0` adds a desktop app shell that can open a local project, inspect shared project status, edit basic config, check ComfyUI connection state, and browse workflows, runs, and assets through a Windows-first Tauri application.

The app should prove the desktop architecture and navigation model without attempting to finish the full gallery, batch UI, or installer work scheduled for later versions.

## Non-Goals

The following remain out of scope for `0.7.0`:

- production installer,
- auto-updater,
- code signing,
- full gallery grid interactions,
- batch task management UI,
- report comparison UI beyond links/summaries,
- custom node editor UI,
- bundled Python runtime,
- full offline packaging,
- ComfyUI workflow graph visual editing,
- running ComfyUI itself,
- replacing Codex as the intelligent workflow/debugging layer.

## Local Tooling

The current development environment has:

- Node `v24.12.0`,
- npm `11.6.2`,
- Rust `1.95.0`,
- Cargo `1.95.0`.

Use npm rather than pnpm because pnpm is not installed in the current environment.

Use Tauri v2 with Vite, React, and TypeScript. The app should be created and maintained under `desktop/` so the Python MCP package remains a separate package.

### Data Bridge

The Tauri Rust backend should not reimplement Comfydex project indexing, path safety, or SQLite logic.

For `0.7.0`, add a Python CLI bridge:

```powershell
python -m comfydex_mcp.desktop_bridge <operation> --workspace <path> [--payload-json <json>]
```

The bridge calls existing public Comfydex modules and returns JSON to stdout.

Supported bridge operations:

- `project_status`
- `reindex_project`
- `get_config`
- `set_config`
- `check_connection`
- `list_workflows`
- `list_runs`
- `search_assets`

The Tauri Rust backend exposes commands that call this bridge with a configured workspace path:

- `app_info`
- `set_workspace`
- `project_status`
- `reindex_project`
- `get_config`
- `set_config`
- `check_connection`
- `list_workflows`
- `list_runs`
- `search_assets`

This keeps the desktop shell aligned with MCP behavior and avoids data-model drift before the 1.0 packaging stage.

### Workspace State

The desktop app stores only the selected workspace path in the Tauri app config directory.

Project data remains in the workspace:

- workflows in `workflows/`,
- runs in `runs/`,
- asset sidecars and reports in `.comfydex/`,
- index database in `.comfydex/comfydex.db`.

The selected workspace path must be explicit. The app should show an empty state until a workspace is selected.

### Frontend API Boundary

`desktop/src/lib/api.ts` owns all Tauri command calls.

It should export typed functions:

- `getAppInfo()`
- `setWorkspace(path)`
- `getProjectStatus()`
- `reindexProject()`
- `getConfig()`
- `setConfig(configPatch)`
- `checkConnection()`
- `listWorkflows()`
- `listRuns()`
- `searchAssets(filters)`

When running under a browser-only Vite dev environment without Tauri, the API should return deterministic development data. This makes frontend build and smoke tests possible without a native window.

### UI Shell

The first screen is the app itself, not a marketing page.

Use a restrained developer-tool layout:

- left navigation rail,
- top project/status bar,
- dense dashboard overview,
- list/detail work areas,
- predictable empty/loading/error states,
- compact controls,
- no hero sections,
- no decorative cards inside cards,
- no gradient/orb decoration.

Primary views:

#### Dashboard

Shows:

- workspace path,
- database path,
- schema version,
- counts for workflows, runs, outputs, assets, batches, errors,
- ComfyUI connection status,
- last reindex timestamp,
- quick actions for reindex and connection check.

#### Workflows

Shows:

- workflow list,
- kind/source/validation status,
- node count,
- model references,
- selected workflow summary.

`0.7.0` does not implement graph editing.

#### Runs

Shows:

- run list,
- workflow name,
- status,
- prompt id,
- output count,
- updated time,
- selected run summary.

#### Assets

Shows:

- search input,
- favorite/rating/tag filters when data exists,
- asset table with filename, workflow, status, rating, favorite, tags,
- selected asset details.

`0.7.0` may use a table-first asset browser. The image gallery grid is scheduled for `0.8.0`.

#### Settings

Shows:

- workspace path,
- ComfyUI base URL,
- request/websocket timeout values,
- redacted headers,
- save button,
- connection check result.

Risky operations such as cleanup confirmation stay out of `0.7.0` UI.

## Visual Design

Use a quiet, dense, technical desktop style:

- neutral dark shell,
- strong text contrast,
- muted accent color for active state,
- small but readable type,
- 8px or smaller radii,
- stable table row heights,
- visible keyboard focus states,
- icons for navigation/actions where useful,
- no marketing hero,
- no oversized headings inside panels.

The UI skill search suggested a highly vibrant showcase pattern. That pattern is rejected for this product because Comfydex is an operational developer tool. Keep the useful part: technical typography and precise dashboard hierarchy.

## Error Handling

Every bridge response should use a stable envelope:

```json
{
  "ok": true,
  "data": {}
}
```

Errors use:

```json
{
  "ok": false,
  "error": {
    "type": "ValueError",
    "message": "..."
  }
}
```

The frontend should show errors inline in the relevant view and keep the app shell usable.

Bridge commands must not expose configured header values. Config responses use existing redaction helpers.

## Security And Path Safety

The desktop app is local-only for `0.7.0`.

Path rules:

- workspace must be an absolute local path,
- workspace must not be redirected,
- config writes must use existing Comfydex config helpers,
- workflow/run/asset reads must use existing safe modules,
- Tauri commands must not accept arbitrary file paths for reads/deletes,
- cleanup confirmation is not exposed in this release.

The Tauri backend should allow only the command set listed in this design.

## Testing

Add tests for:

- Python bridge operations,
- error envelope behavior,
- config redaction,
- project status bridge using a temp workspace,
- asset search bridge using indexed test data,
- frontend build,
- frontend API fallback behavior,
- Tauri Rust command compile or type checks when available.

Release gate includes existing full Python suite plus desktop checks:

```powershell
python -m pytest tests -q
python scripts\validate_plugin.py
python -m json.tool .codex-plugin\plugin.json > $null
python -m json.tool .mcp.json > $null
npm --prefix desktop run build
git diff --check
```

If a full Tauri native build is too slow or requires missing Windows system components, `0.7.0` may gate on `npm --prefix desktop run build` plus Rust unit/type checks. The release notes must state which desktop checks ran.

## Milestones

### Milestone 1: Desktop Bridge

- Add `desktop_bridge.py`.
- Add operations for project status, config, workflows, runs, assets, and connection check.
- Add bridge tests.

Exit criteria:

- bridge returns stable success/error envelopes,
- bridge redacts config headers,
- bridge uses existing project core and config modules.

### Milestone 2: Tauri/Vite Shell Scaffold

- Add `desktop/` npm project.
- Add Tauri v2 config.
- Add React/Vite/TypeScript shell.
- Add browser fallback API.

Exit criteria:

- `npm --prefix desktop run build` passes,
- frontend renders deterministic fallback data outside Tauri.

### Milestone 3: Workbench Views

- Add dashboard, workflows, runs, assets, and settings views.
- Add loading/error/empty states.
- Add navigation and project status bar.

Exit criteria:

- build passes,
- view tests or smoke checks cover all primary views,
- text fits in compact desktop layout.

### Milestone 4: Tauri Commands

- Add Rust commands that call the Python bridge.
- Wire frontend API to Tauri invoke.
- Keep browser fallback for build/test.

Exit criteria:

- command names match API,
- command errors use stable envelopes,
- frontend can call commands through one API layer.

### Milestone 5: Documentation And Release Metadata

- Add desktop app usage docs.
- Update README and Skill.
- Bump versions to `0.7.0`.

Exit criteria:

- docs tests pass,
- version tests pass,
- plugin validation passes.

### Milestone 6: Release Gate

- Run full tests and desktop build checks.
- Perform spec, path safety, UI, bridge, and release metadata reviews.
- Merge to `main`, push, tag `v0.7.0`.
- Start `0.8.0` Gallery And Batch UI planning.
