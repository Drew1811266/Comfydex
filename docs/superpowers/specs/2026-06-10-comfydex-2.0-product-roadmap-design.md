# Comfydex 2.0 Product Roadmap Design

## Goal

Comfydex 2.0 turns ComfyUI from a node-engineering surface into a conversational creation system for ordinary ComfyUI users. A user should be able to describe an image task in natural language, let Codex plan and build the workflow, confirm any model or node installation plan, and then see the workflow loaded into ComfyUI through the Live Bridge.

The 2.0 promise is:

> Say what you want to create; Comfydex understands the task, checks the local ComfyUI environment, builds the workflow, resolves missing requirements, pushes the graph into ComfyUI, and helps recover when it fails.

## Primary User

The primary 2.0 user is a normal ComfyUI user who wants ComfyUI's power without manually understanding every node, model folder, custom node dependency, or workflow compatibility issue.

This user may know image-generation concepts such as text-to-image, image-to-image, ControlNet, LoRA, inpainting, upscaling, or product-photo generation, but should not need to wire the graph manually for common tasks.

Advanced developers remain supported, but they are no longer the main design center for 2.0.

## Product Shape

Comfydex 2.0 is a Codex-driven creative layer over local ComfyUI. It has two primary interaction surfaces:

1. Codex conversation as the main command surface.
2. Comfydex Desktop as a semi-interactive status and confirmation console.

The ComfyUI frontend remains the visual workflow runtime. The Live Bridge is the transport that lets Codex load generated or repaired workflows directly into the running ComfyUI canvas after one bootstrap installation and restart.

## Core Principles

- Prefer working local workflows over abstract explanations.
- Prefer explicit installation plans over silent model or node downloads.
- Make simple requests feel immediate.
- Explain complex plans before executing them.
- Treat the local ComfyUI environment as the source of truth for available nodes, models, versions, memory, and runtime state.
- Use handwritten node semantics for critical workflow reasoning.
- Use ComfyUI `object_info` and filesystem scans for verification, not as the only source of semantic understanding.
- Keep the Desktop console supportive, not a replacement for Codex conversation.

## Architecture

### 1. Node Semantic Registry

The Node Semantic Registry is a handwritten knowledge base for ComfyUI native nodes and selected high-frequency functional node packs.

Each supported node entry must define:

- canonical node type and display names,
- category such as Model, Conditioning, Sampling, Image, Mask, Control, Adapter, or Output,
- user-facing purpose,
- input and output meaning,
- common parameter strategies,
- compatible upstream and downstream node types,
- common failure modes,
- repair hints,
- whether the node is safe for ordinary users,
- whether the node requires external model assets,
- whether the node belongs to a known custom node package.

The registry must not pretend to understand unknown nodes. For unknown nodes, Comfydex may inspect structure through `object_info`, but planning should treat them as unavailable for first-class generation unless a scenario recipe explicitly allows structural passthrough.

### 2. Scenario Recipe Library

The Scenario Recipe Library maps ordinary user requests to workflow plans. Recipes sit above the node registry and reuse node semantics instead of duplicating graph logic per scenario.

2.0 first-class recipes include:

- text-to-image,
- image-to-image and style transfer,
- portrait and character-consistency workflows,
- product and ecommerce image workflows,
- ControlNet workflows for pose, lineart, depth, and similar controls,
- inpainting and mask-driven edits,
- upscaling and detail enhancement,
- background replacement or cleanup when available nodes support it.

Each recipe must define:

- intent signals,
- required and optional capabilities,
- required node semantic groups,
- model asset requirements,
- minimum input artifacts such as prompt, source image, mask, or reference image,
- default parameter policy,
- when to run directly,
- when to present a plan first,
- validation and repair strategy.

### 3. Capability Resolver

The Capability Resolver inspects the user's local ComfyUI environment before a plan becomes executable.

It checks:

- ComfyUI connection,
- Live Bridge status,
- installed node types from `object_info`,
- installed custom node packages where discoverable,
- model folders and model filenames,
- known shared model paths,
- GPU and VRAM from `/system_stats`,
- whether current ComfyUI canvas state can be replaced safely,
- whether a restart or frontend refresh is required.

The resolver returns a capability report with:

- available capabilities,
- missing models,
- missing nodes,
- incompatible versions,
- alternate available paths,
- whether the request can run now,
- whether it needs user confirmation,
- whether it is blocked.

### 4. Install Planner

The Install Planner translates missing requirements into a user-confirmable install plan.

Default mode is conservative:

- No silent large model downloads.
- No silent custom node installation.
- Show source URL or provider, filename, expected size when known, target directory, purpose, and restart requirement.
- Offer an existing local substitute when one is available.
- Record all install actions in a local audit log.
- Verify installation after the user confirms.

Advanced mode can allow semi-automatic installation only for approved sources and whitelisted model or node packages.

### 5. Workflow Planner And Graph Builder

The Workflow Planner converts natural language into a structured workflow plan. It chooses the scenario recipe, resolves environment capability, and decides whether execution is direct or requires a user-visible plan.

The Graph Builder converts the plan into ComfyUI UI workflow JSON. It must produce canvas-positioned graphs that users can inspect and modify in ComfyUI, not only API prompt JSON.

The builder must support:

- stable node IDs,
- readable node titles,
- layout grouping for ordinary users,
- deterministic links,
- optional notes or labels when ComfyUI supports them,
- conversion to API prompt JSON only when execution requires it.

### 6. Execution, Validation, And Repair Loop

Execution happens through the existing MCP tools and the Live Bridge:

1. Build or repair UI workflow.
2. Push it into ComfyUI with Live Bridge.
3. Validate against `object_info`.
4. Convert or build API prompt JSON when a run is requested.
5. Submit only when submit policy allows it.
6. Wait through WebSocket and HTTP fallback.
7. Fetch or register outputs.
8. Reindex project and assets.
9. Diagnose failures and propose a repair.

The repair loop must distinguish:

- missing model,
- missing node,
- invalid parameter,
- bad link type,
- unsupported current environment,
- runtime execution failure,
- out-of-memory or resource failure,
- user cancellation.

### 7. Desktop Semi-Interactive Console

Comfydex Desktop becomes a support console for ordinary users. It should not replace Codex conversation.

2.0 Desktop responsibilities:

- show ComfyUI connection and Live Bridge status,
- show local model and node capability status,
- show pending install plans from Codex,
- let users confirm or reject installation plans,
- show whether a restart or refresh is needed,
- show workflow generation history,
- open or push historical workflows,
- show run failures and repair suggestions,
- retry failed tasks,
- show outputs and asset history.

Desktop should not become a full low-code workflow editor in 2.0.

## Interaction Modes

### Direct Mode

Simple requests can build and push immediately when all requirements are already available and no risky action is needed.

Examples:

- Basic text-to-image with an installed supported model.
- Simple image-to-image with available VAE, sampler, and model path.
- Reopening a previously generated workflow.

Direct mode must not silently overwrite an unsaved canvas unless the user has explicitly allowed force replacement.

### Plan-First Mode

Complex or risky requests must show a plan before execution.

Plan-first mode is required when:

- a model download is needed,
- a custom node install is needed,
- ComfyUI restart or refresh is required,
- current canvas replacement is risky,
- source images or masks need disambiguation,
- the recipe has multiple viable model or graph strategies,
- the task is character consistency, product image generation, ControlNet, inpainting, or another multi-stage workflow.

The plan should explain:

- what workflow will be built,
- which model and node families will be used,
- what is missing,
- what Comfydex proposes to install or substitute,
- whether the result will be pushed to ComfyUI,
- whether execution will run immediately or only load the graph.

## Version Roadmap

### 1.2.0 - Live Bridge Productization

Goal: Make the Live Bridge dependable and visible for ordinary users.

Key deliverables:

- MCP or local tool integration for bridge status, push, reload, and verification.
- Desktop status surface for ComfyUI and Live Bridge.
- Installer upgrade path for the bridge custom node.
- Better error messages for bridge not loaded, frontend not connected, unsaved canvas, and stale client.
- Post-install verification flow.
- Documentation for installing, updating, verifying, and removing the bridge.

Exit criteria:

- A normal user can install the bridge, restart once, verify status, and receive a pushed workflow without reading source code.

### 1.3.0 - Node Semantic Registry Foundations

Goal: Establish the handwritten node knowledge base and use it for planning.

Key deliverables:

- Registry schema and validation.
- Native node semantics for model loading, conditioning, latent, sampling, VAE decode, image loading, image saving, preview, and basic mask/image operations.
- Initial LoRA, ControlNet, IP-Adapter, upscale, and inpaint semantic packs where stable node packages are selected.
- Tests that registry entries match expected ComfyUI object metadata when available.
- Documentation explaining supported first-class nodes and unsupported unknown-node behavior.

Exit criteria:

- Comfydex can explain supported nodes, choose them for basic workflows, and refuse unsupported nodes honestly.

### 1.4.0 - Capability Resolver And Install Planner

Goal: Make Comfydex environment-aware and safe around missing models or nodes.

Key deliverables:

- Local model inventory with model type inference.
- Node inventory from `object_info`.
- Capability report for requested scenarios.
- Missing model and missing node report.
- Conservative installation plan format.
- Desktop confirmation surface for install plans.
- Audit log for install actions.
- Whitelist design for optional advanced semi-automatic install.

Exit criteria:

- Before building a workflow, Comfydex can say whether it will run locally and what must be installed or substituted.

### 1.5.0 - Scenario Recipes For Common Creation Tasks

Goal: Convert ordinary requests into workflow plans for the main user scenarios.

Key deliverables:

- Recipes for text-to-image, image-to-image, style transfer, inpainting, upscaling, ControlNet, portrait, character-consistency, product image, and background replacement.
- Direct vs plan-first decision policy.
- Prompt and parameter extraction from natural language.
- Image input requirement handling.
- Tests for recipe selection and plan generation.

Exit criteria:

- For each first-class scenario, Comfydex can generate a clear executable plan or an install/substitution plan.

### 1.6.0 - UI Graph Builder And Live Canvas UX

Goal: Build user-readable UI workflows, not just executable API prompts.

Key deliverables:

- UI workflow builder for supported recipes.
- Canvas layout rules and grouping conventions.
- Stable node IDs and readable titles.
- Live Bridge push integration.
- History records for generated graphs.
- Desktop view for generated workflow history and push/open actions.

Exit criteria:

- A user can ask for a supported workflow, see a readable graph appear in ComfyUI, and reopen it later from history.

### 1.7.0 - Execution And Repair Loop

Goal: Close the loop from plan to run to output and failure recovery.

Key deliverables:

- Submit policy integration for generated graphs.
- Run orchestration for supported recipe outputs.
- Failure classification for missing assets, invalid links, invalid parameters, execution errors, and resource failures.
- Repair suggestions and one-click retry plan.
- Desktop failure/repair display.
- Regression tests using mocked ComfyUI responses and fixture workflows.

Exit criteria:

- Failed generated workflows produce actionable repairs instead of opaque errors.

### 1.8.0 - Ordinary User Polish

Goal: Make the common path understandable without developer knowledge.

Key deliverables:

- Clear user-facing language for plans, missing items, and errors.
- Better defaults by GPU class and model family.
- Presets for quality, speed, aspect ratio, and style.
- Safer canvas replacement prompts.
- Output asset summaries and comparison support.
- Documentation written for non-developers.

Exit criteria:

- A non-developer can complete common tasks without learning the internal node graph first.

### 2.0.0 - Conversational Workflow System

Goal: Ship the complete ordinary-user workflow system.

2.0 includes:

- Live Bridge productized and verified.
- Handwritten node semantic registry for native and selected high-frequency functional nodes.
- Scenario recipe library for the main creation tasks.
- Capability resolver and install planner.
- Desktop semi-interactive console.
- Direct mode and plan-first mode.
- UI graph generation and live push.
- Execution, validation, output registration, and repair loop.
- Documentation and release checklist for ordinary users.

2.0 exit criteria:

- A normal ComfyUI user can describe any first-class scenario in natural language.
- Comfydex can either build and push a runnable workflow, or explain exactly what is missing and present a safe install/substitution plan.
- Generated workflows are readable in ComfyUI.
- Failed workflows produce actionable diagnosis and repair options.
- Desktop shows status, missing requirements, install plans, history, failures, and outputs.

## Non-Goals For 2.0

- Cloud fleet management.
- Multi-user collaboration server.
- Hosted model marketplace.
- Full visual workflow editor replacement.
- Silent unbounded model downloads.
- Claiming semantic support for arbitrary unknown custom nodes.
- Replacing ComfyUI's canvas or execution engine.

## Risks

### Node Package Drift

High-frequency custom node packages can change APIs. The mitigation is to pin semantic packs to known node type names and validate against current `object_info`.

### Over-Automation

Ordinary users want convenience, but model downloads and node installs can be large or risky. The mitigation is conservative defaults, explicit install plans, whitelists, and audit logs.

### Template Explosion

Hardcoding every scenario as a template would become unmaintainable. The mitigation is a two-layer system: reusable node semantics below scenario recipes.

### Desktop Scope Creep

Desktop could become a full editor. The mitigation is to limit 2.0 Desktop to status, confirmations, history, retries, failures, and outputs.

### Verification Gaps

It is easy to generate a plausible graph that fails locally. The mitigation is capability resolution, `object_info` validation, submit policy, and repair loops before claiming success.

## Success Metrics

- Percentage of first-class scenario requests that produce a runnable workflow without manual node editing.
- Percentage of missing-requirement cases that produce a clear install or substitution plan.
- Number of supported first-class node semantic entries.
- Number of supported scenario recipes.
- Reduction in opaque ComfyUI errors for generated workflows.
- Time from natural language request to workflow visible in ComfyUI.
- Time from failure to actionable repair suggestion.

## Acceptance Criteria

Comfydex 2.0 is complete only when:

- `README.md` and usage docs describe ordinary-user 2.0 behavior.
- The registry validates its first-class node semantics.
- Scenario recipes cover text-to-image, image-to-image, portrait, character-consistency, product image, ControlNet, inpainting, upscaling, and background replacement where dependencies are available.
- Capability reports identify missing models and nodes before execution.
- Install plans require confirmation by default and record audit logs.
- Desktop shows Live Bridge, model, node, install, history, failure, and output status.
- Live Bridge can push generated UI workflows to ComfyUI.
- Generated UI workflows are readable and editable in ComfyUI.
- Execution and repair loops are covered by tests and documented fixtures.
- Release validation checks 2.0 metadata and required docs.
