# Comfydex 0.3 Shared Core Data Layer Design

Date: 2026-06-08
Version target: `0.3.0`
Parent roadmap: `docs/superpowers/specs/2026-06-08-comfydex-1.0-roadmap-design.md`

## Status

This document defines the `0.3.0` release scope. Implementation must follow a
separate plan under `docs/superpowers/plans/` and must pass the release gate in
this document before tagging `v0.3.0`.

## Context

Comfydex `0.2.0` stores durable state as project-local JSON files and output
directories:

- workflows are stored as `workflows/<name>.json`,
- workflow metadata is stored as `workflows/.metadata/<stem>.metadata.json`,
- conversion reports are stored as `workflows/.reports/<stem>.conversion.json`,
- runs are stored as `runs/<run_id>/run.json`,
- run workflow snapshots are stored as `runs/<run_id>/workflow.json`,
- run outputs are stored as files below `runs/<run_id>/outputs/`,
- batch records are stored as `runs/.batches/<batch_id>/batch.json`.

That layout is simple and debuggable, and it must remain readable after `0.3.0`.
However, the 1.0 roadmap needs a shared core that can be used by both the MCP
server and the future Tauri desktop app. The shared core needs stable project
paths, migrations, indexed query surfaces, and recovery behavior that does not
depend on MCP tool internals.

## Goal

Build a shared project data layer that indexes existing workflow, run, output,
and batch records into a workspace-local SQLite database while preserving the
current JSON and filesystem formats as compatibility records.

## Product Outcome

After `0.3.0`, Codex can ask Comfydex for project status and rebuild the project
index. Developers get a durable index that future versions can use for natural
language workflow construction, desktop browsing, asset management, and release
hardening.

The release is successful when:

- existing `0.2.0` workflows, runs, outputs, and batches remain readable,
- a new project database is created safely inside the workspace,
- indexing is idempotent and recoverable,
- corrupt or partially written records are reported without deleting user data,
- MCP exposes project status and project reindex tools,
- the code is organized as a reusable core package rather than MCP-only helpers.

## Non-Goals

`0.3.0` must not implement these features:

- a Tauri desktop UI,
- thumbnails or gallery presentation,
- natural language workflow generation upgrades,
- custom node development loop upgrades,
- cloud sync, team collaboration, or remote project sharing,
- replacing JSON records with SQLite as the only source of truth,
- destructive cleanup of corrupt project files,
- cross-platform packaging beyond preserving Windows-first path safety.

## Existing Behavior To Preserve

The following `0.2.0` behaviors are compatibility requirements:

- `comfy_list_workflows` and `comfy_read_workflow` continue reading workflow
  JSON and metadata from `workflows_dir`.
- `comfy_save_workflow` continues writing JSON workflow files.
- `comfy_submit_workflow`, `comfy_wait_for_run`, `comfy_fetch_outputs`, and
  diagnostic/report tools continue reading and writing `runs_dir` records.
- `comfy_batch_submit` and `comfy_read_batch` continue using
  `runs/.batches/<batch_id>/batch.json`.
- output listing and cleanup continue treating `runs/<run_id>/outputs/` as the
  real file location.
- all current traversal and redirected-path protections remain in force.

`0.3.0` may add index refresh calls after these operations, but index failures
must not corrupt or delete the JSON compatibility records.

## Design Principles

1. Compatibility records first.
   Existing JSON and output files remain human-readable and portable.

2. SQLite as an index, not a replacement.
   SQLite accelerates queries and gives future UI surfaces a consistent data
   model, but project recovery must be possible by reindexing the files.

3. Shared core boundary.
   New data-layer code must not import `server.py` or MCP-specific types.
   `server.py` should call shared core APIs.

4. Idempotent migrations.
   Opening a project can safely run migrations repeatedly.

5. Conservative recovery.
   Corrupt JSON, missing files, redirected paths, and partial writes are reported
   as index errors. The indexer skips unsafe records and preserves the files.

6. Desktop readiness.
   APIs should return plain dictionaries, dataclasses, and JSON-compatible
   values so they can be reused by the future Tauri command bridge.

## Project Layout

The shared project state lives under:

```text
<workspace>/.comfydex/
  comfydex.db
  migrations/
  cache/
  tmp/
```

Initial `0.3.0` implementation only needs to create:

```text
<workspace>/.comfydex/comfydex.db
```

The extra directories are reserved for future thumbnail/cache work and should not
be required by tests unless the implementation uses them.

## Path Model

The core project context must derive:

- `workspace`
- `workflows_dir`
- `runs_dir`
- `state_dir`
- `database_path`

`workspace`, `workflows_dir`, and `runs_dir` come from the current
`ComfydexConfig`. `state_dir` defaults to `<workspace>/.comfydex`.

The database path must stay inside `state_dir`. `state_dir` must stay inside
`workspace` for the default configuration. If a future config option allows a
custom state directory, it must use the same redirected-path checks already used
for workflows, runs, reports, and outputs.

## SQLite Schema

Use the Python standard library `sqlite3`. No new runtime dependency is required.

`schema_migrations` tracks applied migrations:

```sql
CREATE TABLE schema_migrations (
  version INTEGER PRIMARY KEY,
  name TEXT NOT NULL,
  applied_at TEXT NOT NULL
);
```

`project_metadata` tracks database-level state:

```sql
CREATE TABLE project_metadata (
  key TEXT PRIMARY KEY,
  value TEXT NOT NULL,
  updated_at TEXT NOT NULL
);
```

`workflow_records` indexes workflows and metadata:

```sql
CREATE TABLE workflow_records (
  name TEXT PRIMARY KEY,
  path TEXT NOT NULL,
  kind TEXT NOT NULL,
  source TEXT NOT NULL,
  submit_ready INTEGER NOT NULL,
  validation_status TEXT NOT NULL,
  node_count INTEGER NOT NULL,
  node_types_json TEXT NOT NULL,
  model_references_json TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  indexed_at TEXT NOT NULL,
  content_hash TEXT NOT NULL,
  valid_json INTEGER NOT NULL,
  error TEXT
);

CREATE INDEX idx_workflow_records_kind
  ON workflow_records(kind);

CREATE INDEX idx_workflow_records_validation_status
  ON workflow_records(validation_status);
```

`run_records` indexes run records:

```sql
CREATE TABLE run_records (
  run_id TEXT PRIMARY KEY,
  path TEXT NOT NULL,
  workflow_name TEXT,
  prompt_id TEXT,
  client_id TEXT,
  status TEXT NOT NULL,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  output_count INTEGER NOT NULL,
  event_count INTEGER NOT NULL,
  indexed_at TEXT NOT NULL,
  content_hash TEXT NOT NULL,
  valid_json INTEGER NOT NULL,
  error TEXT
);

CREATE INDEX idx_run_records_status
  ON run_records(status);

CREATE INDEX idx_run_records_updated_at
  ON run_records(updated_at);
```

`output_records` indexes real output files:

```sql
CREATE TABLE output_records (
  output_id TEXT PRIMARY KEY,
  run_id TEXT NOT NULL,
  path TEXT NOT NULL,
  filename TEXT NOT NULL,
  type TEXT NOT NULL,
  subfolder TEXT NOT NULL,
  size_bytes INTEGER NOT NULL,
  modified_time REAL NOT NULL,
  downloaded_path TEXT,
  indexed_at TEXT NOT NULL,
  FOREIGN KEY (run_id) REFERENCES run_records(run_id) ON DELETE CASCADE
);

CREATE INDEX idx_output_records_run_id
  ON output_records(run_id);

CREATE INDEX idx_output_records_modified_time
  ON output_records(modified_time);
```

`batch_records` indexes batch records:

```sql
CREATE TABLE batch_records (
  batch_id TEXT PRIMARY KEY,
  path TEXT NOT NULL,
  label TEXT NOT NULL,
  workflow_name TEXT NOT NULL,
  status TEXT NOT NULL,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  run_count INTEGER NOT NULL,
  completed_count INTEGER NOT NULL,
  failed_count INTEGER NOT NULL,
  indexed_at TEXT NOT NULL,
  content_hash TEXT NOT NULL,
  valid_json INTEGER NOT NULL,
  error TEXT
);

CREATE INDEX idx_batch_records_status
  ON batch_records(status);

CREATE INDEX idx_batch_records_updated_at
  ON batch_records(updated_at);
```

`index_errors` records recoverable indexing failures:

```sql
CREATE TABLE index_errors (
  error_id TEXT PRIMARY KEY,
  source_type TEXT NOT NULL,
  source_id TEXT NOT NULL,
  path TEXT NOT NULL,
  message TEXT NOT NULL,
  created_at TEXT NOT NULL
);

CREATE INDEX idx_index_errors_source
  ON index_errors(source_type, source_id);
```

## Data Normalization

All indexed rows must be JSON-compatible when returned through APIs.

### Workflow Index Rows

Workflow rows come from `read_workflow` and `summarize_workflow` where safe.

Fields:

- `name`
- `path`
- `kind`
- `source`
- `submit_ready`
- `validation_status`
- `node_count`
- `node_types`
- `model_references`
- `updated_at`
- `indexed_at`
- `valid_json`
- `error`

If a workflow JSON file is corrupt, the indexer records an `index_errors` row and
does not upsert an invalid workflow row. Existing list tools may continue to show
invalid JSON according to their current behavior.

### Run Index Rows

Run rows come from `read_run` and the run directory.

Fields:

- `run_id`
- `path`
- `workflow_name`
- `prompt_id`
- `client_id`
- `status`
- `created_at`
- `updated_at`
- `output_count`
- `event_count`
- `indexed_at`
- `valid_json`
- `error`

If `created_at` or `updated_at` is missing from old records, use the file
modified time converted to UTC ISO text.

### Output Index Rows

Output rows come from `outputs.list_outputs`. Only files that pass existing safe
output checks are indexed.

`output_id` is a deterministic hash of the resolved absolute file path and the
owning run id. Reindexing must produce the same id for the same file.

### Batch Index Rows

Batch rows come from `read_batch_record`.

`completed_count` and `failed_count` are computed from child run statuses in the
batch JSON record. Unknown or malformed child run entries are ignored for counts
but recorded as an index error.

## Migration Behavior

Opening or reindexing a project must:

1. ensure `<workspace>/.comfydex/` exists,
2. ensure `comfydex.db` exists,
3. enable `PRAGMA foreign_keys = ON`,
4. apply unapplied migrations inside a transaction,
5. write the current schema version to `project_metadata`,
6. return a machine-readable migration report.

Migration report shape:

```json
{
  "database_path": "C:/path/to/workspace/.comfydex/comfydex.db",
  "schema_version": 1,
  "applied_migrations": [
    {"version": 1, "name": "initial_project_index"}
  ]
}
```

## Reindex Behavior

`reindex_project` must:

1. migrate the database,
2. clear previous `index_errors`,
3. scan workflows,
4. scan runs,
5. scan outputs,
6. scan batches,
7. delete stale rows whose source files no longer exist,
8. write `project_metadata.last_reindexed_at`,
9. return counts and errors.

Reindex report shape:

```json
{
  "status": "completed",
  "database_path": "C:/path/to/workspace/.comfydex/comfydex.db",
  "schema_version": 1,
  "counts": {
    "workflows": 3,
    "runs": 5,
    "outputs": 8,
    "batches": 1,
    "errors": 0
  },
  "errors": []
}
```

If recoverable indexing errors occur, `status` is `completed_with_errors`. If the
database cannot be opened or migrated, `status` is `failed` and no destructive
filesystem action is allowed.

## Write-Time Index Refresh

The first implementation should support both explicit reindexing and targeted
write-time refresh.

After these MCP operations write compatibility records, they should refresh the
affected index rows:

- `comfy_save_workflow`
- `comfy_build_workflow`
- `comfy_patch_workflow`
- `comfy_import_ui_workflow`
- `comfy_convert_ui_to_api`
- `comfy_submit_workflow`
- `comfy_wait_for_run`
- `comfy_fetch_outputs`
- `comfy_batch_submit`

The refresh must be best-effort. If it fails, the original MCP operation should
return its primary result and include an index warning only where that can be
done without changing established return contracts. For operations whose return
contract should stay unchanged, the next `comfy_project_status` call must expose
the index error.

If this proves too broad during implementation, explicit reindexing is the
minimum release requirement. Any deferred write-time refresh must be documented
in the `0.3.0` release notes.

## Public Core API

Create a shared core package under:

```text
src/comfydex_mcp/core/
```

Proposed modules:

- `core/project.py`
  - `ProjectContext`
  - `project_context_from_config(config)`
  - `project_status(context)`
- `core/database.py`
  - SQLite connection setup
  - transaction helpers
  - migration application
- `core/schema.py`
  - schema version constants
  - migration SQL
- `core/indexer.py`
  - `reindex_project(context)`
  - targeted indexing helpers for workflows, runs, outputs, and batches
- `core/store.py`
  - query/upsert helpers
  - count helpers
  - stale-row cleanup

The implementation can merge modules if the final files stay small and clear,
but MCP tools must call the shared core API instead of duplicating database SQL
inside `server.py`.

```json
  "runs_dir": "C:/path/to/workspace/runs",

  "database_path": "C:/path/to/workspace/.comfydex/comfydex.db",

  "counts": {
    "workflows": 3,
    "runs": 5,
    "outputs": 8,
    "batches": 1,
    "errors": 0
  },

}
```

This tool must not contact ComfyUI.

### `comfy_reindex_project`

Inputs:

```json
{
  "include_outputs": true
}
```

This tool must not contact ComfyUI.

## Risks And Mitigations

### Risk: SQLite Index Diverges From JSON

Mitigation:

- preserve explicit reindexing,
- treat JSON/filesystem as source-of-recovery,
- write `last_reindexed_at`,
- expose index error counts in project status.

### Risk: Database Write Failure Breaks Existing MCP Flows

Mitigation:

- keep write-time refresh best-effort,
- do not delete or rewrite compatibility records during indexing,
- return index warnings only where contracts can tolerate them.

### Risk: Path Safety Regression

Mitigation:

- reuse existing safe path helpers,
- add tests for redirected outputs and unsafe database paths,
- never follow redirected output directories.

### Risk: Scope Creep Into Desktop Or Gallery

Mitigation:

- keep `0.3.0` limited to project state and indexing,
- defer thumbnails, gallery filters, visual UI, and asset tagging to later
  versions.
