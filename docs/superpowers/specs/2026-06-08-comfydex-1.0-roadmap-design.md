# Comfydex 1.0 Roadmap Design

## Status

Comfydex `0.2.0` is released on `main` and tagged as `v0.2.0`.

This document defines the long-running roadmap from `0.3.0` to `1.0.0`.
Each minor version must be planned, implemented, reviewed, merged to `main`,
pushed to GitHub, and tagged before the next minor version begins.

## Product Definition

Comfydex `1.0.0` is a Windows-first local ComfyUI developer toolchain made of:

- a Codex plugin and MCP server,
- a shared local project core,
- a Tauri desktop app,
- a common data model for workflows, runs, outputs, assets, batches, reports, and custom node work.

The 1.0 release must be usable for a developer who runs ComfyUI locally and wants Codex to help build, validate, run, debug, and manage ComfyUI work.

## Non-Goals For 1.0

These are intentionally out of scope for `1.0.0`:

- team collaboration,
- cloud-first or multi-tenant operation,
- full macOS/Linux desktop support,
- ComfyUI frontend extension integration,
- remote server fleet management,
- marketplace or model distribution features.

Remote ComfyUI URLs may continue to work through existing config, but local Windows workflows are the release target.

## 1.0 Capability Targets

### Codex Developer Toolchain

Codex should be able to:

- inspect existing workflows,
- generate complex workflow drafts from natural language,
- validate workflows against API shape and ComfyUI object metadata,
- repair generated workflows when errors are mechanically resolvable,
- explain unresolved gaps clearly,
- submit safe validated workflows,
- wait for runs,
- fetch outputs,
- diagnose failures,
- compare runs and workflows.

### Automatic Run Policy

Comfydex may automatically submit and run a workflow when all are true:

- the workflow is a single-run request,
- API validation passes,
- object metadata validation passes or is unavailable but no known missing node is detected,
- no output or workflow file would be overwritten unexpectedly,
- no batch or high-cost run is requested,
- no required model or custom node is missing.

Comfydex must require explicit confirmation before running when any high-risk condition exists:

- missing model,
- missing node,
- uncertain required input,
- file overwrite,
- batch submission,
- high step count or large image dimensions,
- cleanup deletion,
- unknown validation state.

### Custom Node Developer Loop

Custom node support should include:

- workspace-local scaffolding,
- package inspection,
- mapping validation,
- class contract validation,
- example input generation,
- node contract tests,
- ComfyUI type validation for common types,
- import checks,
- deterministic docs,
- actionable error-location guidance.

### Asset Library And Run Management

Comfydex should provide a local asset system that can:

- index runs,
- index output files,
- preserve workflow snapshots,
- store prompt/model metadata,
- generate thumbnails and sidecar metadata when possible,
- search and filter by workflow, prompt text, model, status, tags, rating, and date,
- compare outputs and runs,
- mark favorites,
- cleanup safely through dry-run and confirmation,
- export reports.

### Desktop App

The desktop app target is Windows-first Tauri.

The desktop app should share the same project data as the MCP server and expose:

- project open/create,
- local ComfyUI connection status,
- config management,
- workflow browser,
- run browser,
- asset gallery,
- batch task view,
- reports and comparisons,
- custom node package view,
- safety confirmation surfaces for risky actions.

Codex remains responsible for intelligent workflow construction and debugging. The desktop app is responsible for visual management and explicit user controls.

## Architecture Direction

### Shared Core

From `0.3.0` onward, persistent state should move toward a shared core package that both MCP tools and desktop app bindings can use.

The shared core owns:

- project layout,
- local database,
- file sidecars,
- migrations,
- indexing,
- repositories/services for workflows, runs, outputs, batches, reports, and custom nodes,
- path safety boundaries.

### Storage Model

Use a hybrid storage model:

- SQLite for indexes, search metadata, tags, ratings, status, and relationships.
- JSON files for workflow snapshots and existing compatibility records.
- Filesystem directories for outputs, thumbnails, reports, and example assets.

The storage layer must preserve compatibility with the existing `workflows/` and `runs/` directories.

### API Boundary

MCP tools should call shared services instead of directly manipulating scattered files.

The desktop app should not call private MCP internals. It should use a stable local API or command bridge over the same shared core.

## Per-Version Execution Rule

For every version from `0.3.0` through `1.0.0`:

1. Write a version design document under `docs/superpowers/specs/`.
2. Write a detailed implementation plan under `docs/superpowers/plans/`.
3. Implement tasks using test-first changes where code behavior changes.
4. Run targeted tests for the version.
5. Run the full suite:

```powershell
python -m pytest -q
```

6. Run plugin validation:

```powershell
python scripts/validate_plugin.py
```

7. Validate JSON manifests:

```powershell
python -m json.tool .codex-plugin/plugin.json > $null
python -m json.tool .mcp.json > $null
```

8. Run a spec compliance review and code quality review.
9. Resolve all Critical and Important findings.
10. Bump version files and release notes.
11. Merge to `main`.
12. Push `main` to GitHub.
13. Create and push the version tag, for example `v0.3.0`.
14. Start planning the next minor version automatically.

## Review Gate

Each version must pass these review categories:

- spec compliance,
- path safety,
- data migration safety,
- backward compatibility,
- deterministic output,
- clear tool errors,
- test coverage,
- documentation consistency,
- release metadata consistency.

Critical and Important findings block release.

## Backward Compatibility

The 1.0 roadmap must preserve these existing 0.2 behaviors unless explicitly replaced by a compatible migration:

- existing workflow files remain readable,
- existing run records remain readable,
- output paths remain safe,
- configured headers remain redacted,
- existing MCP tool names remain available or receive documented compatibility aliases,
- plugin manifest remains valid,
- tests for 0.1 and 0.2 behaviors continue passing.

## 0.3.0 Starting Point

The next version is `0.3.0: Shared Core Data Layer`.

Its design must answer:

- exact project directory layout,
- SQLite schema,
- migration strategy from existing `runs/` and `workflows/`,
- repository/service APIs,
- MCP tool compatibility plan,
- desktop-app-readiness boundaries,
- indexing and re-indexing behavior,
- corruption and partial-write recovery,
- tests and release gate.

## 1.0 Completion Criteria

Comfydex reaches `1.0.0` only when:

- all roadmap versions `0.3.0` through `0.9.0` are released,
- MCP tools and desktop app share one project core,
- a Windows user can install/run the toolchain locally,
- Codex can generate, validate, safely run, and fetch a non-trivial workflow from a natural-language request,
- high-risk actions require confirmation,
- custom node development has a tested validation loop,
- asset gallery and run management are usable from the desktop app,
- full tests, plugin validation, JSON manifest validation, and release reviews pass,
- README and usage docs describe the final 1.0 behavior accurately.
