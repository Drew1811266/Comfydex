# Comfydex 1.9 2.0 Readiness Gate Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship Comfydex `1.9.0` with a machine-readable 2.0 readiness gate that audits first-class scenarios, recipe coverage, semantic coverage, UI graph buildability, install-plan safety, repair-loop availability, desktop visibility, and release documentation before the final 2.0 release.

**Architecture:** Add a focused `readiness.py` service that composes existing registries instead of duplicating workflow logic. Expose that service through MCP and the desktop bridge, surface the report in Comfydex Desktop as a status/coverage console, then make the release validator and docs require the 1.9 readiness contract. The report may say `needs_work`; 1.9 is the evidence gate that makes the remaining 2.0 gaps explicit.

**Tech Stack:** Python 3.11+, FastMCP server tools, existing Comfydex registries (`recipes`, `node_semantics`, `generation`, `ui_graphs`, `capabilities`, `repairs`), Tauri desktop bridge, React/TypeScript desktop UI, pytest, npm, cargo.

---

## Scope

1.9 is a readiness release, not the final 2.0 release. It must not claim unsupported workflows are fully buildable. It must produce a stable report that tells Codex and users which first-class scenarios are ready, partially covered, or still blocked.

The 1.9 exit line is:

- `comfy_20_readiness_report` returns a JSON-compatible report with version `1.9.0`.
- All 2.0 first-class scenarios are represented in the report.
- Existing runnable recipe scenarios prove plan and UI graph buildability with deterministic dry-run inputs.
- Missing first-class scenarios produce explicit gaps and suggested next implementation work.
- MCP, desktop bridge, and Desktop UI can display the same readiness data.
- Release docs and validation describe the gate and do not overstate 2.0 completion.

Add report functions:

```python
def list_first_class_scenarios() -> list[dict[str, Any]]:
    return [scenario.to_dict() for scenario in FIRST_CLASS_SCENARIOS]


def build_20_readiness_report() -> dict[str, Any]:
    scenarios = [_scenario_readiness(scenario) for scenario in FIRST_CLASS_SCENARIOS]
    ready_count = sum(1 for scenario in scenarios if scenario["status"] == "ready")
    blocked = [scenario for scenario in scenarios if scenario["status"] != "ready"]
    acceptance = _acceptance_criteria(scenarios)
    status = "ready_for_2_0" if not blocked and all(item["status"] == "ready" for item in acceptance) else "needs_work"
    next_steps = _next_steps(scenarios, acceptance)
    return {
        "readiness_version": READINESS_VERSION,
        "status": status,
        "summary": {
            "scenario_count": len(scenarios),
            "ready_count": ready_count,
            "needs_work_count": len(scenarios) - ready_count,
        },
        "scenarios": scenarios,
        "acceptance_criteria": acceptance,
        "next_steps": next_steps,
    }
```

Implement `_scenario_readiness`, `_buildability_for_recipe`, `_acceptance_criteria`, and `_next_steps` so that ready recipes prove:

The scenario status rules are:

- `ready`: at least one mapped recipe exists, dry-run plan has no missing information, UI graph build status is `valid`, and semantic coverage is `supported`.
- `partial`: a mapped recipe exists but dry-run buildability or semantic coverage is incomplete.
- `missing_recipe`: no mapped recipe exists.
