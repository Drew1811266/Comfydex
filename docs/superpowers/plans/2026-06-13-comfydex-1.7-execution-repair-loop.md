# Comfydex 1.7 Execution And Repair Loop Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship Comfydex `1.7.0` with generated-workflow execution failure classification, actionable repair plans, safe retry operations, and desktop failure/repair visibility.

**Architecture:** Add a focused repair planning layer on top of existing run diagnostics. Keep diagnosis read-only, represent repair plans as structured data with conservative retry operations, then surface those plans through MCP, the Python desktop bridge, and the desktop Runs view. Integrate failed `comfy_generate_run_fetch` responses with the same repair plan shape so users get the same recovery language from automation, direct MCP tools, and desktop.

**Tech Stack:** Python 3.11, pytest, FastMCP, existing `diagnostics.py`, `runs.py`, `server.py`, `desktop_bridge.py`, React/Tauri desktop bridge, TypeScript, Markdown docs.

---

## Scope Boundary

1.7 implements:

- Failure classification for generated and submitted workflow runs.
- Repair plans for missing models, missing nodes, invalid parameters, invalid links, runtime execution errors, resource failures, fetch failures, and missing outputs.
- Safe retry operations for low-risk cases such as output fetch retry and explicit resubmit of an existing workflow.
- MCP tools to plan and retry run repairs.
- Integration of repair plans into `comfy_generate_run_fetch` failed responses.
- Desktop bridge operations for run repair plans and retries.
- Desktop Runs view failure/repair display and retry action.
- Docs, tests, and version metadata for `1.7.0`.

1.7 does not implement:

- Automatic model downloads.
- Automatic custom node installation.
- Arbitrary graph rewriting from opaque ComfyUI errors.
- Full visual repair editing in Desktop.
- GPU telemetry collection beyond classifying error text already present in run events/history.
- Silent resubmission without explicit confirmation.

Create `src/comfydex_mcp/repairs.py` with:

- `classify_run_failure(diagnosis: dict[str, Any], *, stage: str | None = None, error: str | None = None) -> dict[str, Any]`.
- `build_run_repair_plan(run_id: str, diagnosis: dict[str, Any], *, workflow_name: str | None = None, stage: str | None = None, error: str | None = None) -> dict[str, Any]`.

Class priorities:

1. `missing_model` when `missing_model_references` is non-empty.
2. `missing_node` when `missing_node_types` is non-empty.
3. `missing_outputs` when signal `missing_outputs` is present.
4. `resource_failure` when summary/error text contains `out of memory`, `cuda out of memory`, `vram`, `allocation`.
5. `invalid_parameter` when text contains `invalid`, `bad value`, `not in list`, `expected`.
6. `invalid_link` when text contains `link`, `type mismatch`, `cannot connect`.
7. `fetch_failure` when `stage == "fetch"`.
8. `execution_error` when signal `execution_error`, `history_failed`, or status `failed` is present.
9. `unknown_failure` fallback.

Plan action kinds:

- `select_model` for missing models.
- `install_node` for missing nodes.
- `fetch_outputs` for missing outputs and fetch failures.
- `reduce_workload` for resource failures.
- `adjust_parameter` for invalid parameters.
- `inspect_links` for invalid links.
- `inspect_history` for execution errors and unknown failures.

Retry operation:

- `fetch_outputs` without confirmation for `missing_outputs` and `fetch_failure`.
- `resubmit_workflow` with confirmation for `resource_failure`, `invalid_parameter`, `invalid_link`, and `execution_error`.
- unsupported for missing models, missing nodes, and unknown failures.

Implementation rules:

- `comfy_plan_run_repair` reads the run, workflow snapshot, optional object info, diagnosis, builds repair plan, appends history with `status="planned"`, and returns `{"status": "planned", "diagnosis": ..., "repair_plan": ...}`.
- `comfy_retry_run_repair` first builds a fresh plan.
- If retry unsupported, return `{"status": "blocked", "repair_plan": ..., "next_actions": [...]}`.
- If retry requires confirmation and `confirm` is false, return `{"status": "requires_confirmation", "repair_plan": ...}`.
- For `fetch_outputs`, call `comfy_fetch_outputs(run_id)` and append history with `status="retried"`.
- For `resubmit_workflow`, call `comfy_submit_workflow(record["workflow_name"], run_label=f"{run_id}-repair")` and append history with `status="retried"`.
